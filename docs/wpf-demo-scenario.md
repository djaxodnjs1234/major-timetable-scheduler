# WPF 시간표 스케줄러 시연 시나리오

## 시연 목표

WPF 프로토타입이 교과목·교수·강의실 정보를 관리하고, 제약 조건을 바탕으로 시간표 후보를 생성하며, 미리보기에서 뒤로 돌아와도 교과목 메타데이터가 유지되는 흐름을 보여준다. 이후 선택한 시간표를 수동 편집하고 저장하는 전체 사용 흐름까지 설명한다.

## 오프닝 멘트

"이 프로그램은 전공 시간표를 자동 생성하고, 생성된 후보를 미리 본 뒤 필요한 경우 수동으로 조정할 수 있는 WPF 기반 시간표 스케줄러입니다. 오늘은 정보 입력, 자동 생성, 미리보기, 뒤로가기 안정성, 수동 편집, 저장 흐름을 순서대로 보여드리겠습니다."

## 1. 교과목 정보 관리 화면 소개

1. 앱을 실행하고 정보 입력 화면으로 이동한다.
2. 교과목 탭을 선택한다.
3. 교과목 목록에서 다음 항목을 설명한다.
   - 교과목 코드와 이름
   - 분반 수
   - 담당교수
   - 이수구분
   - 학년
   - 주당 수업시간
   - 블록 구조
   - 불가 강의실
   - 공동 담당교수
   - 시간 고정 요약

멘트:
"교과목 목록은 같은 교과목의 여러 분반을 하나의 그룹으로 묶어서 보여줍니다. 담당교수나 이수구분처럼 모든 분반이 공유하는 조건은 한 번만 편집하고, 고정 시간처럼 분반마다 달라질 수 있는 조건은 분반별로 관리합니다."

## 2. 분반이 여러 개인 교과목 편집

1. 분반이 2개인 교과목을 펼친다.
2. 담당교수, 이수구분, 시수, 블록 구조, 불가 강의실, 공동 담당교수가 그룹 단위 조건임을 보여준다.
3. 고정 시간은 분반별로 설정할 수 있음을 보여준다.
4. 담당교수를 한 번 변경하고 저장한다.
5. A/B 분반 모두 같은 담당교수로 유지되는지 확인한다.

멘트:
"분반이 여러 개인 교과목도 담당교수를 분반마다 반복 입력하지 않습니다. 그룹에서 한 번 저장하면 공통 조건이 모든 분반에 동기화됩니다."

## 3. 단일 분반 교과목 안정성 확인

1. 분반이 1개인 교과목을 펼친다.
2. 담당교수가 정상 표시되는 것을 확인한다.
3. 시간표 생성 탭으로 이동한다.
4. 시간표 후보를 생성한다.
5. 미리보기 화면으로 이동한다.
6. 뒤로가기를 눌러 교과목 정보 관리 화면으로 돌아온다.
7. 단일 분반 교과목의 담당교수가 그대로 유지되는지 확인한다.

멘트:
"이전에는 미리보기로 이동하며 이전 ComboBox가 해제될 때 담당교수 ID가 빈 값으로 바뀌는 문제가 있었습니다. 지금은 완료 저장 시 화면 편집 객체와 작업공간 데이터를 분리해, 화면이 사라져도 담당교수 값이 유지됩니다."

## 4. 시간표 자동 생성

1. 자동 생성 탭으로 이동한다.
2. 생성 설정을 설명한다.
   - 생성할 후보 개수
   - 제한 시간
   - 선호 조건 사용 여부
   - Cross 설정
3. 공통 고정 강의실을 쓰는 두 과목을 Cross로 묶으려 하면 `IE-039`가 표시되고 저장되지 않는 것을 확인한다.
4. 시간표 자동 생성을 실행한다.
5. 후보가 생성될 때까지 진행 상태를 보여준다.

멘트:
"자동 생성은 반드시 지켜야 하는 하드 제약을 만족하는 후보를 먼저 찾고, 이후 선호 조건 점수를 기준으로 후보를 정렬합니다."

## 5. 생성 결과 미리보기

1. 미리보기 화면으로 이동한다.
2. 후보 카드 중 하나를 선택한다.
3. 통합 시간표를 보여준다.
4. 시간표 칸에 표시되는 정보를 설명한다.
   - 교과목명과 분반
   - 이수구분과 담당교수
   - 강의실
5. 후보가 여러 개라면 다른 후보도 선택해 비교한다.

멘트:
"미리보기에서는 여러 후보를 비교하면서 수업 배치, 교수, 강의실 정보를 한눈에 확인할 수 있습니다."

## 6. 미리보기 뒤로가기 안정성 확인

1. 미리보기 화면에서 뒤로가기를 누른다.
2. 교과목 정보 관리 화면으로 돌아온다.
3. 다음 값이 유지되는지 확인한다.
   - 단일 분반 교과목의 담당교수
   - 다중 분반 교과목의 담당교수
   - 이수구분
4. 필요하면 조건을 수정한 뒤 다시 생성할 수 있음을 설명한다.

멘트:
"시간표 생성은 보통 한 번에 끝나지 않고, 생성 결과를 보고 다시 조건을 조정하는 반복 작업입니다. 그래서 미리보기에서 뒤로 돌아와도 입력 데이터가 안전하게 유지되는 것이 중요합니다."

## 7. 수동 편집으로 넘기기

1. 다시 미리보기 화면으로 이동한다.
2. 사용할 후보를 선택한다.
3. 수동 편집 화면으로 이동한다.
4. 수업 하나를 다른 시간이나 강의실로 이동해본다.
5. 충돌이 있다면 검증 메시지가 표시되는 것을 보여준다.
6. 가능한 배치로 조정한 뒤 저장한다.

멘트:
"자동 생성 결과를 그대로 쓰지 않고, 사용자가 필요한 부분만 직접 조정할 수 있습니다. 수동 편집에서도 교수·강의실·시간 충돌을 확인하면서 안전하게 수정합니다."

## 8. 저장된 시간표 확인

1. 저장 후 시간표 선택 화면으로 돌아온다.
2. 저장된 시간표를 선택한다.
3. 저장된 시간표 미리보기를 확인한다.
4. 저장 시점의 교과목·교수·강의실 정보가 함께 보존된다는 점을 설명한다.

멘트:
"저장된 시간표는 당시의 교과목, 교수, 강의실 정보를 스냅샷으로 함께 보관합니다. 그래서 이후 입력 데이터가 바뀌더라도 저장된 시간표는 생성 당시 정보로 다시 확인할 수 있습니다."

## 강조할 핵심 기능

- 교과목 공통 조건은 그룹 단위로 한 번만 편집한다.
- 시간 고정처럼 분반마다 다른 조건은 분반별로 관리한다.
- 자동 생성 후 미리보기에서 뒤로 돌아와도 담당교수와 이수구분이 유지된다.
- 미리보기와 수동 편집은 같은 교과목 메타데이터를 사용한다.
- 저장된 시간표는 생성 당시의 정보 스냅샷을 보존한다.

## 예상 질문과 답변

### 담당교수를 왜 분반마다 따로 입력하지 않나요?

이 프로토타입에서는 담당교수를 교과목 공통 조건으로 보고 있습니다. 같은 교과목의 A/B 분반이 같은 담당교수를 공유하는 경우가 많기 때문에, 한 번만 편집하고 모든 분반에 동기화합니다.

### 담당교수가 사라지던 문제의 원인은 무엇이었나요?

완료 저장 시 화면의 교과목 편집 객체를 작업공간에 그대로 넣어, 화면이 사라질 때 ComboBox의 빈 값이 실제 담당교수 ID까지 바꾸는 문제가 있었습니다. 이제 작업공간에는 별도 복사본을 저장하므로 화면 재구성이 실제 생성 데이터에 영향을 주지 않습니다.

### DTO를 전부 다시 설계해야 했나요?

아닙니다. 장기적으로는 교과목 그룹 편집 DTO가 더 깔끔하지만, 이번 문제는 WPF ComboBox 바인딩 경계에서 발생한 값 소실이라 최소 수정으로 해결했습니다.

### 자동 생성과 수동 편집은 어떻게 연결되나요?

자동 생성이 가능한 후보를 만들고, 미리보기에서 후보를 고른 뒤 수동 편집으로 넘깁니다. 수동 편집에서는 사용자가 배치를 조정하되, 충돌 검사를 통해 잘못된 이동을 막습니다.
