# 협업 규칙

## 브랜치 전략

- `main` — 항상 동작하는 안정 버전. **직접 커밋 금지**
- `feature/기능명` — 기능 개발
- `fix/버그명` — 버그 수정

```bash
# 작업 시작할 때
git checkout main
git pull origin main
git checkout -b feature/내기능
```

---

## 커밋 메시지 규칙

```
기능 추가: 시간표 저장 기능 구현
버그 수정: 블록 이동 시 충돌 감지 오류 수정
UI 수정: 정보입력 버튼 명칭 변경
리팩토링: ManualEditViewModel 분리
```

- 커밋 단위는 **하나의 작업 단위**로
- 완성되지 않은 코드는 커밋하지 않기

---

## PR 규칙

### PR 올리기 전

```bash
git fetch origin
git rebase origin/main   # main 최신 반영 후 올리기
git push origin feature/내기능
```

### PR 제목

```
[기능] 시간표 저장 기능 구현
[수정] 블록 이동 충돌 감지 오류 수정
[UI]   정보입력 버튼 명칭 변경
[리팩] ManualEditViewModel 분리
```

### PR 본문 템플릿

```
## 작업 내용
- 변경한 내용을 간단히 설명

## 변경 파일
- 주요 변경 파일 나열

## 테스트
- 테스트한 항목 체크
  - [ ] 정상 동작 확인
  - [ ] 예외 케이스 확인
```

### PR 리뷰
- merge 전 **상대방 리뷰 필수**
- 충돌은 **PR 올린 사람**이 직접 해결
- 본인 PR 직접 merge 금지

---

## Merge 후 브랜치 삭제

GitHub 설정에서 자동 삭제 활성화:
> Settings → General → **Automatically delete head branches** ✅

로컬 브랜치는 직접 정리:
```bash
git branch -d feature/기능명   # 로컬 브랜치 삭제
git fetch --prune              # 원격 삭제된 브랜치 로컬 정리
```

---

## 금지 사항

- `main`에 직접 `push` ❌
- `git push --force` (main 브랜치에) ❌
- 리뷰 없이 본인 PR 직접 merge ❌

---

## 전체 플로우 요약

```
main pull → 브랜치 생성 → 작업 → 커밋
→ main rebase → push → PR 생성 → 리뷰
→ merge → 브랜치 삭제
```
