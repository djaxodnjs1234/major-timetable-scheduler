# 프로그램 동작과 시간표 생성 흐름 쉬운 설명

이 문서는 웹으로 옮기기 전에 현재 WPF 프로그램이 어떻게 움직이는지, 특히 시간표 생성기가 어떤 순서로 생각하고 답을 찾는지 이해하기 위한 문서입니다.

목표는 코드를 모르는 사람도 다음 질문에 답할 수 있게 되는 것입니다.

- 사용자가 입력한 과목, 교수, 강의실 정보는 어디로 가는가?
- 시간표 생성 버튼을 누르면 내부에서 어떤 일이 일어나는가?
- 왜 어떤 시간표는 실패하고, 어떤 시간표는 여러 개 만들어지는가?
- 만들어진 시간표는 어떻게 점수가 매겨지고 화면에 보이는가?
- 수동편집은 생성된 시간표와 어떤 관계인가?

영어 약어와 어려운 말은 되도록 줄였습니다. 꼭 필요한 이름은 코드 위치를 찾기 쉽도록 괄호 안에만 적었습니다.

## 1. 한 줄 요약

이 프로그램은 사용자가 입력한 과목, 교수, 강의실, 제약조건을 한 번 복사한 뒤, "절대 어기면 안 되는 규칙"을 모두 만족하는 시간표를 찾고, 그중 "되도록 좋으면 좋은 조건"이 좋은 시간표를 골라 보여주는 프로그램입니다.

## 2. 전체 흐름 한눈에 보기

```text
1. 사용자가 정보 입력
   ↓
2. 입력 오류 검사
   ↓
3. 생성 가능성 검사
   ↓
4. 현재 입력 데이터를 생성용 사본으로 복사
   ↓
5. 시간표 생성기가 규칙을 모두 붙인 문제를 만듦
   ↓
6. 먼저 선호 조건의 좋은 기준값을 찾음
   ↓
7. 여러 번 시도해서 서로 다른 시간표 후보를 모음
   ↓
8. 후보마다 점수를 다시 계산
   ↓
9. 좋은 순서로 정렬해서 결과 화면에 표시
   ↓
10. 사용자가 저장하거나 수동편집으로 이동
```

핵심은 4번입니다. 생성 버튼을 누르면 현재 입력 상태를 그대로 복사합니다. 생성 도중 사용자가 화면을 바꾸거나, 나중에 DB가 바뀌어도 이미 생성 중인 계산에는 영향을 주지 않게 하기 위해서입니다.

## 3. 프로그램에 들어가는 주요 데이터

### 3.1 과목

과목에는 이런 정보가 있습니다.

- 과목 이름
- 학년
- 분반
- 주당 수업 시간
- 담당 교수
- 팀티칭 교수
- 블록 구조
- 고정 시간
- 고정 강의실
- 사용 불가 강의실

예를 들어 `자료구조`가 주당 3시간이고 블록 구조가 `2+1`이면, 프로그램은 이 과목을 "연속 2시간짜리 한 덩어리"와 "1시간짜리 한 덩어리"로 배치하려고 합니다.

### 3.2 교수

교수에는 이런 정보가 있습니다.

- 교수 이름
- 수업할 수 없는 시간
- 수업할 수 없는 강의실

예를 들어 김교수가 월요일 1, 2교시가 불가능하면, 김교수가 담당하는 과목은 월요일 1, 2교시에 들어갈 수 없습니다.

### 3.3 강의실

강의실에는 이런 정보가 있습니다.

- 강의실 이름
- 실습실 여부
- 수용 인원

현재 시간표 생성 규칙에서 가장 중요한 것은 강의실 이름입니다. 같은 시간에 같은 강의실을 두 과목이 동시에 쓸 수 없기 때문입니다.

### 3.4 Cross 묶음

Cross는 두 과목을 특별한 방식으로 엮는 설정입니다.

예를 들어 A 과목과 B 과목이 각각 2분반이면:

```text
A-01은 B-02와 같은 시간
A-02는 B-01과 같은 시간
```

이런 식으로 서로 엇갈려 배치됩니다. 같은 학년 과목은 보통 시간이 겹치면 안 되지만, Cross로 묶인 과목은 일부러 겹치게 만들 수 있습니다.

### 3.5 재수강 고려

재수강 고려를 켜면, 윗학년 학생이 아랫학년 전공을 다시 들어야 하는 상황을 조금 더 안전하게 만듭니다.

중요한 점은 "모든 분반이 안전해야 한다"가 아닙니다. "적어도 하나의 분반은 안전해야 한다"입니다.

## 4. 화면별 역할

### 4.1 시간표 선택 화면

처음 저장된 시간표를 고르거나 새 시간표를 만들러 가는 입구입니다.

### 4.2 정보 입력 화면

과목, 교수, 강의실, Cross, 재수강 옵션, 선호 조건을 입력하는 곳입니다.

사용자가 시간표 생성 버튼을 누르면 이 화면에서 생성 흐름이 시작됩니다.

### 4.3 결과 화면

시간표 생성기가 만든 여러 후보를 카드 형태로 보여줍니다.

각 후보에는 점수, 요일별 밀도, 작은 미리보기, 통합 시간표가 붙습니다.

### 4.4 수동편집 화면

생성된 시간표 하나를 사람이 직접 조정하는 곳입니다.

최근 웹 이전 설계에서는 수동편집은 현재 DB를 계속 따라가지 않고, 생성 당시 사본을 기준으로 독립적으로 편집하는 방향입니다.

## 5. 생성 버튼을 누르면 먼저 하는 검사

시간표 생성기는 바로 계산을 시작하지 않습니다. 먼저 빠르게 잡을 수 있는 문제를 검사합니다.

### 5.1 입력 오류 검사

예를 들면:

- 과목 이름이 비어 있음
- 교수 이름이 비어 있음
- 강의실 이름이 비어 있음
- 고정 시간 개수가 주당 시수와 맞지 않음
- 블록 구조 합계가 주당 시수와 다름
- 존재하지 않는 강의실을 고정 강의실로 선택함

이런 문제는 생성기를 돌려도 답이 나올 수 없거나, 사용자가 먼저 고쳐야 하는 정보입니다.

### 5.2 생성 조건 오류 검사

입력값 자체는 있어도, 시간표를 만들기 어려운 조합인지도 미리 봅니다.

예를 들면:

- 고정 시간이 점심시간에 들어감
- 대학원 과목 시간이 너무 많음
- 교수 불가능 시간이 고정 시간과 충돌함
- Cross로 묶은 과목들의 분반 수가 다름
- 같은 학년 과목이 너무 많아서 넣을 칸이 부족함

이 검사는 완벽한 증명은 아니지만, 사용자에게 "왜 안 되는지"를 빨리 알려주는 역할을 합니다.

## 6. 생성용 사본을 만드는 이유

검사를 통과하면 프로그램은 현재 입력 데이터를 생성용으로 복사합니다.

코드에서는 `SchedulingSnapshot`이라는 흐름입니다.

여기서 사본을 만드는 이유는 세 가지입니다.

1. 계산 중 화면 데이터가 바뀌어도 흔들리지 않게 하려고
2. 분반 묶음 값을 생성에 맞게 정리하려고
3. 저장된 결과가 어떤 입력에서 나왔는지 보존하려고

예를 들어 같은 과목의 여러 분반이 있으면, 과목명, 학년, 시수, 블록 구조 같은 공통 정보는 대표 분반 기준으로 맞춰집니다. 하지만 교수는 분반마다 다를 수 있으므로 그대로 보존합니다.

## 7. 생성기에게 시간표란 무엇인가

사람이 보면 시간표는 표입니다.

하지만 생성기에게 시간표는 수많은 "예/아니오 선택"입니다.

예를 들어 이런 질문을 아주 많이 만듭니다.

```text
자료구조가 월요일 3교시에 D327을 쓰는가? 예/아니오
자료구조가 월요일 4교시에 D327을 쓰는가? 예/아니오
캡스톤디자인이 목요일 6교시에 D330을 쓰는가? 예/아니오
```

이 예/아니오 값들을 모두 잘 골라서 규칙을 만족하면 하나의 시간표가 됩니다.

## 8. 두 종류의 선택값

현재 생성기는 크게 두 종류의 선택값을 씁니다.

### 8.1 강의실까지 포함한 선택

코드에서는 `x`라고 부릅니다.

뜻은:

```text
이 과목이 이 요일, 이 교시, 이 강의실을 쓰는가?
```

강의실 충돌을 검사할 때 필요합니다.

예:

```text
x[자료구조, 월요일, 3교시, D327] = 1
```

이면 자료구조가 월요일 3교시에 D327을 쓴다는 뜻입니다.

### 8.2 시간만 보는 선택

코드에서는 `y`라고 부릅니다.

뜻은:

```text
이 과목이 이 요일, 이 교시에 수업이 있는가?
```

강의실은 보지 않습니다.

교수 충돌, 학년 충돌, 분반 충돌은 강의실보다 시간이 중요하므로 `y`를 봅니다.

### 8.3 왜 둘 다 필요한가

다중강의실 과목 때문입니다.

예를 들어 캡스톤디자인이 D327과 D330을 동시에 쓴다면, 강의실 기준으로는 2개를 씁니다.

하지만 시간표 시간 기준으로는 수업 1개입니다.

그래서 프로그램은 이런 관계를 강제로 맞춥니다.

```text
그 시간에 잡힌 강의실 수 = 필요한 강의실 수 × 시간 사용 여부
```

말로 풀면:

- 일반 과목은 한 시간에 방 하나를 쓰면 수업 있음
- 다중강의실 과목은 한 시간에 필요한 방을 모두 써야 수업 있음

## 9. OR-Tools는 내부에서 대략 어떻게 움직이는가

OR-Tools는 시간표를 아는 프로그램이 아닙니다.

OR-Tools는 이런 말을 이해합니다.

```text
이 값은 0 또는 1 중 하나다.
이 값들의 합은 1 이하여야 한다.
이 값들의 합은 정확히 3이어야 한다.
이 조건이 켜지면 저 값도 켜져야 한다.
가능한 답 중에서 이 숫자가 가장 작으면 좋겠다.
```

즉 OR-Tools는 "시간표 전문가"가 아니라 "조건을 만족하는 숫자 조합을 찾는 도구"입니다.

우리가 하는 일은 시간표 문제를 숫자 조합 문제로 바꿔 주는 것입니다.

### 9.1 OR-Tools에 넣는 세 가지

OR-Tools에 보통 세 가지를 넣습니다.

1. 선택값
2. 반드시 지켜야 하는 규칙
3. 좋으면 좋은 기준

예를 들어:

```text
선택값:
자료구조가 월요일 3교시에 D327을 쓰는가? 0 또는 1

반드시 지켜야 하는 규칙:
월요일 3교시에 D327을 쓰는 과목 수는 1개 이하여야 한다.

좋으면 좋은 기준:
월요일 오전 수업 수는 적을수록 좋다.
```

### 9.2 OR-Tools가 답을 찾는 감각

정확한 내부 구현은 매우 복잡하지만, 감각적으로는 이렇게 이해하면 됩니다.

```text
1. 아직 정하지 않은 선택값이 많다.
2. 어떤 값을 1로 해본다.
3. 그 선택 때문에 불가능해지는 다른 선택들을 지운다.
4. 모순이 생기면 방금 선택을 버리고 다른 길을 본다.
5. 모순이 없으면 계속 더 정한다.
6. 모든 규칙을 만족하면 시간표 후보 하나를 얻는다.
7. 더 좋은 후보가 있는지 계속 찾아본다.
```

예를 들어 `자료구조 = 월요일 3교시 D327`을 켰다고 해봅니다.

그러면 OR-Tools는 연결된 규칙 때문에 이런 것들을 자동으로 못 하게 줄입니다.

- 같은 시간 D327에 다른 과목을 넣는 선택
- 같은 교수의 다른 수업을 월요일 3교시에 넣는 선택
- 같은 학년의 다른 과목을 월요일 3교시에 넣는 선택

이렇게 하나를 정할 때마다 주변 가능성이 줄어듭니다.

### 9.3 가지치기

OR-Tools가 빠른 이유 중 하나는 모든 경우를 무식하게 끝까지 다 만들지 않기 때문입니다.

중간에 이미 말이 안 되는 길이면 바로 버립니다.

예를 들어 어떤 과목이 주당 3시간이어야 하는데, 남은 가능한 칸이 2개밖에 없으면 그 길은 더 볼 필요가 없습니다.

이렇게 불가능한 길을 빨리 버리는 것을 여기서는 쉬운 말로 "가지치기"라고 부르겠습니다.

### 9.4 좋은 답 찾기

절대 규칙만 넣으면 OR-Tools는 "되는 시간표"를 찾습니다.

그런데 우리는 "되는 시간표 중에서 더 좋은 시간표"를 원합니다.

그래서 좋으면 좋은 기준도 숫자로 만듭니다.

예:

```text
월요일 오전/금요일 오후에 들어간 수업 수를 줄이고 싶다.
```

이 기준을 OR-Tools에는 이렇게 넣습니다.

```text
이 나쁜 시간에 들어간 선택값들의 합을 작게 만들어라.
```

즉 선호 조건도 결국 숫자 계산으로 바꿔서 넣습니다.

### 9.5 OR-Tools가 해주는 일과 우리가 해주는 일

OR-Tools가 해주는 일:

- 0/1 선택값을 조합해 본다.
- 규칙 때문에 불가능한 선택을 줄인다.
- 모순이 생기면 다른 길을 찾는다.
- 가능한 답을 찾는다.
- 좋은 기준을 작게 또는 크게 만드는 답을 찾는다.

우리가 해주는 일:

- 과목, 요일, 교시, 강의실을 0/1 선택값으로 만든다.
- 교수 충돌, 강의실 충돌, 학년 충돌 같은 규칙을 수식으로 바꾼다.
- 블록 구조, 고정 시간, 다중강의실 같은 우리 학과 규칙을 넣는다.
- 월요일 오전 피하기 같은 선호 조건을 숫자로 바꾼다.
- 나온 0/1 결과를 다시 사람이 보는 시간표로 바꾼다.

한 줄로 말하면:

```text
우리는 시간표 문제를 OR-Tools가 풀 수 있는 숫자 문제로 번역했고,
OR-Tools는 그 숫자 문제의 답을 찾아준다.
```

## 10. 우리가 OR-Tools를 어떻게 활용했는가

이 프로젝트는 OR-Tools를 "시간표 배치 계산기"처럼 씁니다.

하지만 계산기에 그냥 "좋은 시간표 만들어줘"라고 말하지 않습니다.

아래처럼 아주 구체적인 문제지를 만들어 넣습니다.

### 10.1 모든 가능한 칸을 먼저 만든다

예를 들어 과목이 30개, 요일이 5개, 교시가 13개, 강의실이 10개라면, 이런 질문이 매우 많이 생깁니다.

```text
과목 A가 월요일 1교시에 R1을 쓰는가?
과목 A가 월요일 1교시에 R2를 쓰는가?
과목 A가 월요일 2교시에 R1을 쓰는가?
...
```

이 질문 하나하나가 OR-Tools의 선택값입니다.

### 10.2 강의실 선택값과 시간 선택값을 연결한다

강의실까지 보는 값은 `x`입니다.

시간만 보는 값은 `y`입니다.

우리는 둘을 이렇게 연결합니다.

```text
그 시간에 선택된 강의실 수 = 필요한 강의실 수 × 그 시간 수업 여부
```

이 연결 덕분에 다중강의실 과목도 깨지지 않습니다.

### 10.3 블록 시작 선택값도 만든다

블록 수업은 단순히 칸을 아무렇게나 2개 고르면 안 됩니다.

2시간 수업이면 붙어 있어야 합니다.

그래서 프로그램은 이런 선택값도 만듭니다.

```text
자료구조의 첫 번째 블록이 월요일 3교시에 시작하는가?
자료구조의 첫 번째 블록이 화요일 6교시에 시작하는가?
```

이 시작 선택값이 켜지면, 그 블록 안의 실제 수업 칸들이 함께 켜집니다.

예:

```text
2시간 블록이 월요일 3교시에 시작한다
→ 월요일 3교시 수업 있음
→ 월요일 4교시 수업 있음
```

### 10.4 절대 규칙을 모두 붙인다

이제 선택값들이 만들어졌으니, 규칙을 붙입니다.

예:

```text
같은 강의실 같은 시간의 선택값 합은 1 이하
같은 교수 같은 시간의 선택값 합은 1 이하
과목별 전체 수업 시간 합은 주당 시수와 같아야 함
점심시간 선택값은 0
고정 시간 선택값은 1
```

OR-Tools 입장에서는 이것이 시간표 규칙이 아니라 숫자 규칙입니다.

하지만 우리가 잘 번역했기 때문에, 숫자 규칙을 만족하는 답이 곧 시간표 규칙을 만족하는 답이 됩니다.

### 10.5 선호 조건은 단계별로 쓴다

우리는 선호 조건을 한꺼번에 섞지 않고 단계별로 봅니다.

```text
1단계: 월요일 오전/금요일 오후를 최대한 피하기
2단계: 그 수준을 유지하면서 교수 수업 요일 줄이기
3단계: 앞의 수준을 유지하면서 블록 간격 좋게 하기
4단계: 그 조건 안에서 여러 후보 모으기
```

이렇게 하면 어떤 선호 조건이 먼저 중요한지 흐름이 분명해집니다.

### 10.6 여러 답을 얻기 위해 여러 번 시도한다

OR-Tools는 보통 좋은 답 하나를 찾는 데 강합니다.

그런데 우리 프로그램은 사용자에게 여러 후보를 보여줘야 합니다.

그래서 마지막 단계에서 여러 번 시도합니다.

시도할 때마다 찾는 순서를 조금씩 흔듭니다.

그 결과 서로 다른 시간표가 나오면 후보 목록에 추가합니다.

이미 본 시간표와 완전히 같으면 중복으로 버립니다.

### 10.7 OR-Tools 결과를 다시 시간표로 바꾼다

OR-Tools가 돌고 나면 우리는 `x` 선택값 중 1인 것들을 읽습니다.

예:

```text
x[자료구조, 월요일, 3교시, D327] = 1
x[자료구조, 월요일, 4교시, D327] = 1
```

이것을 다시 이런 배정 줄로 바꿉니다.

```text
자료구조 / 월요일 / 3교시 / D327
자료구조 / 월요일 / 4교시 / D327
```

그리고 화면에서는 연속된 줄을 묶어서 하나의 수업 카드처럼 보여줍니다.

## 11. OR-Tools를 썼다고 해서 끝이 아닌 이유

OR-Tools 사용 자체는 핵심 도구 선택입니다.

하지만 실제 제품에서 어려운 부분은 다음입니다.

- 어떤 선택값을 만들지
- 어떤 규칙을 절대 규칙으로 넣을지
- 어떤 규칙을 선호 조건으로 둘지
- 다중강의실을 어떻게 깨지지 않게 할지
- Cross와 재수강 같은 학과 특수 규칙을 어떻게 번역할지
- 답이 없을 때 사용자에게 어떤 이유를 알려줄지
- 결과를 사람이 보는 표로 어떻게 다시 묶을지
- 수동편집 후에도 충돌을 어떻게 검사할지

그래서 "OR-Tools를 썼다"는 말은 시작점입니다.

이 프로젝트의 실제 가치는 우리 학과 시간표 규칙을 OR-Tools가 풀 수 있는 문제로 자세히 번역해 둔 데 있습니다.

## 12. 절대 어기면 안 되는 규칙

이 문서에서는 쉬운 말로 "절대 규칙"이라고 부르겠습니다.

절대 규칙을 하나라도 어기면 그 시간표는 답이 아닙니다.

### 12.1 같은 강의실 동시 사용 금지

같은 요일, 같은 교시, 같은 강의실에는 한 과목만 들어갈 수 있습니다.

### 12.2 같은 교수 동시 수업 금지

같은 교수는 같은 시간에 두 수업을 할 수 없습니다.

팀티칭 교수도 포함합니다.

### 12.3 교수 불가능 시간 금지

교수가 수업할 수 없다고 표시한 시간에는 그 교수가 맡은 과목을 넣지 않습니다.

### 12.4 주당 시수 맞추기

주당 3시간 과목은 정확히 3시간 들어가야 합니다.

다중강의실 과목은 강의실 줄 수가 더 많아지지만, 시간 수는 여전히 주당 시수와 맞아야 합니다.

### 12.5 블록 구조 지키기

`2+1` 과목은 연속 2시간 한 번, 1시간 한 번으로 들어가야 합니다.

`3` 과목은 연속 3시간 한 번으로 들어갑니다.

### 12.6 같은 과목 분반끼리 시간 겹침 금지

자료구조 1분반과 자료구조 2분반은 같은 시간에 열리면 안 됩니다.

### 12.7 같은 학년의 다른 과목끼리 시간 겹침 금지

2학년 자료구조와 2학년 운영체제는 같은 시간에 열리면 안 됩니다.

단, Cross로 묶은 과목은 예외입니다.

### 12.8 점심시간 금지

현재 WPF는 5교시를 점심시간으로 보고 수업을 넣지 않습니다.

웹 이전 설계에서는 이 부분을 학과별 설정으로 바꾸기로 했습니다.

### 12.9 고정 시간 지키기

어떤 과목이 "화요일 3교시 고정"이면 반드시 그 시간에 들어가야 합니다.

중요한 점:

- 고정 시간은 시간만 고정합니다.
- 강의실은 별도 규칙으로 정합니다.

### 12.10 고정 강의실과 불가 강의실

과목에 고정 강의실이 있으면 그 방만 씁니다.

과목에 불가 강의실이 있으면 그 방은 쓰지 않습니다.

고정 강의실이 여러 개면 그중 하나를 고르는 것이 아니라 모두 동시에 씁니다.

### 12.11 같은 교수의 두 분반은 붙여 배치

같은 과목이 정확히 2분반이고, 두 분반의 교수진과 블록 구조가 같으면, 가능한 경우 두 분반을 같은 날 앞뒤로 붙입니다.

예:

```text
자료구조 1분반: 월 1~2
자료구조 2분반: 월 3~4
```

### 12.12 Cross 과목 엇갈림

Cross로 묶인 두 과목은 분반을 엇갈려 같은 시간에 둡니다.

학생들이 선택할 수 있는 조합을 만들기 위한 규칙입니다.

### 12.13 재수강 안전 분반

재수강 고려를 켜면, 재수강 대상 과목의 여러 분반 중 적어도 하나는 윗학년 전공 시간과 겹치지 않게 합니다.

### 12.14 2시간 블록 시작 교시 제한

현재 WPF에서는 2시간 블록이 시작할 수 있는 시간이 정해져 있습니다.

예:

```text
1~2, 3~4, 6~7, 8~9, 10~11, 12~13
```

그래서 2시간 수업이 2~3교시처럼 어중간하게 시작하지 않습니다.

### 12.15 같은 과목의 여러 블록은 다른 요일

`2+1` 과목이면 2시간 블록과 1시간 블록은 서로 다른 요일에 들어갑니다.

### 12.16 자동 배정 과목의 강의실 일관성

강의실을 자동으로 고르는 과목은 같은 과목의 여러 분반, 여러 블록이 같은 강의실을 쓰도록 묶습니다.

## 13. 되도록 좋으면 좋은 조건

절대 규칙은 반드시 지켜야 합니다.

반면 선호 조건은 "가능하면 좋다"입니다.

현재 대표 선호 조건은 세 가지입니다.

### 13.1 월요일 오전, 금요일 오후 피하기

월요일 오전과 금요일 오후에 수업이 몰리는 것을 줄입니다.

### 13.2 교수 수업 요일을 3일 이하로 선호

교수 한 명의 수업이 너무 여러 요일에 흩어지지 않도록 합니다.

### 13.3 한 과목의 여러 블록 사이 간격 선호

같은 과목의 여러 블록은 너무 붙거나 너무 멀지 않게, 보통 2일 차이를 좋게 봅니다.

## 14. 생성기는 왜 여러 단계를 도는가

처음부터 모든 선호 조건을 한꺼번에 좋게 만들려고 하면 기준이 흐려질 수 있습니다.

그래서 현재 프로그램은 순서를 정해서 봅니다.

```text
1단계: 월요일 오전/금요일 오후를 얼마나 피할 수 있는지 먼저 확인
2단계: 1단계 수준을 유지하면서 교수 요일을 얼마나 줄일 수 있는지 확인
3단계: 앞의 수준을 유지하면서 과목 블록 간격을 얼마나 좋게 할 수 있는지 확인
4단계: 위 기준을 만족하는 여러 시간표를 모음
```

이 방식은 "먼저 중요한 선호를 최대한 맞추고, 그다음 선호를 본다"는 뜻입니다.

## 15. 각 단계에서 실제로 하는 일

### 15.1 1단계

월요일 오전과 금요일 오후를 피하는 정도를 봅니다.

가장 좋은 값을 찾으면 그 값을 기록합니다.

### 15.2 2단계

1단계에서 찾은 좋은 수준을 유지하면서, 교수 수업 요일을 줄입니다.

예를 들어 1단계에서 "불리한 시간 2개까지가 최선"이라고 나오면, 2단계에서는 그보다 나빠지지 않게 묶어 둡니다.

### 15.3 3단계

앞의 두 기준을 유지하면서, 과목 블록 간격을 좋게 만듭니다.

### 15.4 4단계

이제 선호 조건 기준이 정해졌으니, 그 기준 안에서 여러 번 시도합니다.

매번 씨앗값을 바꿔서 조금씩 다른 후보를 찾습니다.

씨앗값은 "찾는 순서를 살짝 흔드는 숫자" 정도로 이해하면 됩니다.

## 16. 중복 시간표는 어떻게 거르는가

생성기가 여러 번 시도하면 같은 시간표가 또 나올 수 있습니다.

그래서 프로그램은 시간표를 문자열처럼 줄 세워 비교합니다.

비교에 쓰는 내용은:

- 과목
- 요일
- 교시
- 강의실

이 조합이 완전히 같으면 같은 시간표로 보고 한 번만 보관합니다.

## 17. 생성 결과는 어떤 모양으로 나오나

생성 결과는 이런 줄들의 목록입니다.

```text
자료구조, 월요일, 3교시, D327
자료구조, 월요일, 4교시, D327
자료구조, 수요일, 6교시, D327
캡스톤디자인, 목요일, 6교시, D327
캡스톤디자인, 목요일, 6교시, D330
```

여기서 캡스톤디자인처럼 같은 요일, 같은 교시에 방이 두 개면 다중강의실 수업입니다.

화면에서는 이런 줄들을 다시 묶어서 하나의 수업 카드처럼 보여줍니다.

## 18. 점수는 왜 다시 계산하는가

생성기는 선호 조건을 단계별로 쓰고, 마지막에는 여러 후보를 모읍니다.

결과 화면에서는 사용자에게 좋은 순서로 보여줘야 하므로 각 후보의 점수를 다시 계산합니다.

현재 점수는 세 부분입니다.

```text
월요일 오전/금요일 오후 피하기 점수
교수 요일 집중 점수
과목 블록 간격 점수
```

각 점수는 보통 0에서 1 사이로 봅니다.

전체 점수는 세 점수를 더해서 만듭니다.

## 19. 결과 화면은 무엇을 보여주는가

결과 화면은 생성된 시간표 후보들을 보여줍니다.

주요 화면 요소:

- 후보 카드
- 종합 점수
- 요일별 수업 밀도
- 작은 시간표 미리보기
- 통합 시간표
- 학년별 시간표
- 강의실별 시간표
- 교수별 시간표

사용자는 후보 하나를 고르고 저장하거나 수동편집으로 보낼 수 있습니다.

## 20. 저장은 무엇을 저장하는가

시간표를 저장할 때는 단순히 배정 줄만 저장하면 부족합니다.

왜냐하면 나중에 DB가 바뀌어도 저장된 시간표는 당시 과목명, 교수명, 강의실 정보를 유지해야 하기 때문입니다.

그래서 저장할 때는:

- 시간표 배정 결과
- 그 시간표를 만들 때 사용한 과목/교수/강의실 사본
- Cross, 재수강 같은 설정

을 함께 보관합니다.

이것이 최근 문서에서 말한 스냅샷 개념입니다.

## 21. 수동편집은 생성기와 어떻게 다른가

생성기는 "조건에 맞는 시간표를 새로 찾는 곳"입니다.

수동편집은 "이미 만들어진 시간표 하나를 사람이 고치는 곳"입니다.

수동편집에서 중요한 차이는:

- 생성기를 다시 돌리지 않습니다.
- 사람이 직접 수업 블록을 옮깁니다.
- 옮긴 결과가 충돌을 일으키는지만 검사합니다.
- 저장 전에는 빠진 블록이나 충돌이 없는지 확인합니다.

웹 이전 설계에서는 수동편집이 현재 DB를 따라가지 않고, 저장된 시간표 사본을 기준으로 움직입니다.

## 22. 수동편집에서 충돌 검사는 어떻게 하는가

수동편집은 생성기를 돌리는 것이 아닙니다.

대신 현재 배정 결과를 보고 문제가 있는지 검사합니다.

예를 들면:

- 같은 강의실에 두 과목이 있는가?
- 같은 교수가 두 과목을 동시에 맡았는가?
- 점심시간에 수업이 들어갔는가?
- 고정 시간 과목이 다른 시간으로 이동했는가?
- 고정 강의실 과목이 다른 방을 쓰는가?
- 같은 학년 수업이 겹쳤는가?
- 2시간 블록이 허용되지 않은 시간에 시작했는가?

이 검사는 "이미 놓인 시간표가 말이 되는가"를 확인하는 역할입니다.

## 23. 실패 상태는 어떻게 이해하면 좋은가

시간표 생성 결과 상태는 대략 이렇게 보면 됩니다.

### 23.1 답을 찾음

조건을 만족하는 시간표를 찾은 상태입니다.

### 23.2 답이 없음

주어진 조건으로는 시간표를 만들 수 없다는 뜻입니다.

예:

- 같은 시간에 반드시 고정된 과목이 너무 많음
- 교수 불가능 시간이 너무 많음
- 강의실 후보가 없음
- 같은 학년 과목이 너무 많아 넣을 칸이 부족함

### 23.3 시간이 부족함

답이 없는지, 찾지 못한 것인지 확실히 말하기 전에 제한 시간이 끝난 상태입니다.

이 경우 제한 시간을 늘리거나 조건을 줄이고 다시 시도할 수 있습니다.

### 23.4 취소됨

사용자가 취소한 상태입니다.

## 24. 웹으로 옮길 때 특히 기억할 점

### 24.1 생성과 수동편집을 구분하기

생성은 현재 입력과 제약조건을 바탕으로 새 시간표를 만드는 일입니다.

수동편집은 이미 만들어진 시간표 사본을 고치는 일입니다.

두 화면은 서로 자동으로 영향을 주면 안 됩니다.

### 24.2 점심시간은 고정 상수로 두지 않기

현재 WPF는 5교시 점심을 강하게 가정합니다.

웹에서는 학과마다 다를 수 있으므로 점심 정책을 설정값으로 빼야 합니다.

### 24.3 검증 데이터는 스냅샷 안에 들어 있어야 함

수동편집이 DB를 보지 않더라도 검증은 해야 합니다.

그러려면 과목, 교수, 강의실, 생성 당시 정책이 수동편집 사본 안에 함께 있어야 합니다.

### 24.4 다중강의실은 선택지가 아니라 동시 점유

`D327, D330`이면 둘 중 하나를 쓰는 것이 아닙니다.

두 방을 같은 시간에 동시에 쓰는 것입니다.

## 25. 아주 짧은 발표용 설명

누군가에게 이 프로그램을 1분 안에 설명해야 한다면 이렇게 말하면 됩니다.

```text
이 프로그램은 과목, 교수, 강의실 정보를 받아서 시간표를 자동으로 만드는 프로그램입니다.
먼저 이름 누락, 고정 시간 오류, 강의실 후보 부족 같은 문제를 검사합니다.
그다음 현재 입력을 사본으로 복사해서 생성기가 흔들리지 않게 합니다.
생성기는 과목이 어느 요일, 어느 교시, 어느 강의실에 들어갈지를 수많은 예/아니오 선택으로 바꿉니다.
그리고 강의실 중복 금지, 교수 중복 금지, 주당 시수, 블록 구조, 학년 중복 금지 같은 절대 규칙을 모두 붙입니다.
그 위에 월요일 오전 피하기, 교수 요일 줄이기 같은 선호 조건을 단계별로 좋게 맞춥니다.
마지막에는 여러 번 시도해서 서로 다른 후보를 모으고, 점수를 매겨 좋은 순서로 결과 화면에 보여줍니다.
사용자는 그중 하나를 저장하거나 수동편집에서 직접 조정할 수 있습니다.
```

## 27. 꼭 알아야 하는 말 풀이

### 스냅샷

특정 시점의 데이터를 통째로 복사해 둔 사본입니다.

시간표 생성에서는 "생성 당시 과목, 교수, 강의실, 설정, 배정 결과"를 보존하는 뜻으로 씁니다.

### 블록

수업이 연속으로 붙어 있는 덩어리입니다.

2시간 블록은 예를 들어 1~2교시처럼 붙어 있는 수업입니다.

### 절대 규칙

반드시 지켜야 하는 규칙입니다.

하나라도 어기면 시간표가 답이 될 수 없습니다.

### 선호 조건

가능하면 좋게 맞추고 싶은 조건입니다.

어겼다고 바로 시간표가 틀린 것은 아니지만 점수가 낮아집니다.

### 후보

생성기가 찾아낸 시간표 하나입니다.

여러 후보 중 점수가 좋은 것들이 결과 화면에 보입니다.

### 다중강의실

한 수업이 같은 시간에 여러 강의실을 동시에 쓰는 경우입니다.

## 28. 이해 확인용 퀴즈

### 문제

1. 시간표 생성 버튼을 누르면 생성기가 바로 계산을 시작하지 않고 먼저 하는 두 가지 검사는 무엇인가요?

2. 생성 전에 현재 입력 데이터를 사본으로 복사하는 이유를 두 가지 이상 설명해보세요.

3. 생성기가 쓰는 두 종류의 선택값은 각각 무엇을 뜻하나요?

4. 다중강의실 과목에서 `D327, D330`이 지정되었다면, 둘 중 하나를 고르는 것인가요? 아니면 둘 다 쓰는 것인가요?

5. 같은 교수의 두 과목이 같은 시간에 들어가면 어떤 절대 규칙을 어기는 것인가요?

6. `2+1` 블록 구조는 실제 시간표에서 어떤 모양으로 배치되어야 하나요?

7. 절대 규칙과 선호 조건의 차이를 설명해보세요.

8. 현재 생성기가 선호 조건을 한꺼번에 처리하지 않고 여러 단계로 나누는 이유는 무엇인가요?

9. 생성 결과가 여러 개 나왔을 때, 결과 화면은 무엇을 기준으로 좋은 후보를 위에 보여주나요?

10. 수동편집은 생성기와 무엇이 다른가요?

11. 수동편집이 현재 DB를 직접 보지 않아도 충돌 검사를 할 수 있으려면 무엇이 함께 저장되어 있어야 하나요?

12. 점심시간 정책을 웹으로 옮길 때 `period == 5`를 코드 곳곳에 직접 쓰면 왜 위험한가요?

### 정답 예시

1. 입력 오류 검사와 생성 조건 오류 검사입니다.

2. 계산 중 데이터가 흔들리지 않게 하기 위해서, 분반 정보를 생성에 맞게 정리하기 위해서, 저장된 결과가 어떤 입력에서 나왔는지 보존하기 위해서입니다.

3. 하나는 "과목이 특정 요일, 교시, 강의실을 쓰는가"이고, 다른 하나는 "과목이 특정 요일, 교시에 수업이 있는가"입니다.

4. 둘 다 동시에 씁니다. 다중강의실은 여러 방 중 선택이 아니라 동시 점유입니다.

5. 같은 교수 동시 수업 금지 규칙을 어깁니다.

6. 연속 2시간짜리 수업 한 번과 1시간짜리 수업 한 번으로 배치되어야 합니다. 두 블록은 현재 규칙상 서로 다른 요일에 들어갑니다.

7. 절대 규칙은 반드시 지켜야 하며 어기면 답이 아닙니다. 선호 조건은 가능하면 좋게 맞추는 조건이며 점수에 영향을 줍니다.

8. 먼저 중요한 선호 기준을 최대한 맞추고, 그 수준을 유지한 채 다음 선호를 보기 위해서입니다.

9. 월요일 오전/금요일 오후 회피, 교수 요일 집중, 블록 간격 같은 점수를 계산해서 전체 점수가 높은 후보를 위에 보여줍니다.

10. 생성기는 조건에 맞는 시간표를 새로 찾고, 수동편집은 이미 만들어진 시간표 하나를 사람이 직접 고칩니다. 수동편집은 보통 생성기를 다시 돌리지 않습니다.

11. 과목, 교수, 강의실, Cross, 재수강 설정, 생성 당시 시간 정책, 제약 설정, 실제 배정 결과가 스냅샷으로 함께 저장되어 있어야 합니다.

12. 학과마다 점심시간 규칙이 달라질 수 있기 때문입니다. 어떤 학과는 4교시 금지, 어떤 학과는 5교시 금지, 어떤 학과는 4/5 중 하나만 비우기일 수 있으므로 설정값을 기준으로 판단해야 합니다.
