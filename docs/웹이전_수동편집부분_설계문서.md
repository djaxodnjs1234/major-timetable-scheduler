# 문서 요약

이 문서는 WPF의 수동편집 기능을 웹으로 마이그레이션할 때 필요한 화면 분리, 데이터 모델, 편집 세션, 과목/교수/팀티칭/강의실 변경, 검증, 저장, 내보내기 규칙을 정리한 설계 문서입니다.

가장 큰 변경점은 수동편집 화면이 더 이상 현재 DB나 제약사항 편집 화면에 의존하지 않는다는 점입니다. 시간표가 한 번 생성되면 그 결과는 `ManualEditSession` 스냅샷으로 고정되고, 이후 DB의 과목/교수/강의실/제약조건을 수정해도 이미 열린 수동편집 화면에는 자동 반영되지 않습니다. 제약사항 편집 화면은 다음 시간표 생성을 위한 설정 화면이고, 수동편집 화면은 특정 생성 결과를 독립적으로 편집하는 화면입니다.

웹 수동편집에서는 기존 배정 이동뿐 아니라 과목 추가, 과목 수정, 과목 삭제를 지원해야 합니다. 과목 수정에는 담당 교수 변경, 팀티칭 교수 변경, 강의실 변경, 다중 강의실 설정 변경이 포함됩니다. 모든 변경은 DB 원본이 아니라 수동편집 세션 내부 스냅샷에 적용되며, 저장할 때는 이 편집된 시간표 사본을 별도 결과물로 저장합니다.

# Web Migration: Manual Edit Session, Snapshot, and Editing Rules

This document defines the web manual-edit feature. It complements
`web_migration_constraints_and_solver_policy.md`, but it has a different
responsibility:

- constraint/settings screens define data for future generation.
- manual edit edits one already-generated timetable snapshot.

Manual edit must not be a live view of the current database.

## 1. Core Product Decision

### 1.1 Current Problem

The old behavior is too dependent on the live DB/workspace. If a user changes
course, professor, room, or constraint settings elsewhere, manual edit can be
affected indirectly.

That is unsafe for the web product because:

- users expect a generated timetable to remain stable.
- manual editing is often a review/finalization workflow.
- later DB edits may be intended for the next generation, not the current
  timetable.
- audit/history becomes unclear if saved timetable records silently change.

### 1.2 New Rule

Once a timetable is generated, create a manual-edit snapshot.

```text
generated result + generation input snapshot + generation policy snapshot
  -> ManualEditSession
```

After that:

- manual edit reads only the session snapshot.
- manual edit writes only to the session snapshot.
- DB changes do not update an open or saved manual-edit session.
- constraint setting changes do not update an open or saved manual-edit
  session.
- saving manual edit stores a new timetable version or updates that timetable's
  saved snapshot, depending on product UX.

### 1.3 Screen Separation

The web app must separate:

1. Constraint/data editing screen
   - edits master data or generation settings.
   - affects future solver runs.
   - does not mutate existing manual-edit sessions.

2. Manual-edit screen
   - edits one timetable snapshot.
   - may contain local course/professor/room changes.
   - does not write those changes back to the master DB unless the user uses a
     separate explicit "apply to master data" feature.

Do not implement implicit two-way synchronization between these screens.

## 2. Manual Edit Data Model

Concrete example:

- `docs/manual_edit_snapshot_example.json` shows a full saved session payload
  with `baseline`, `working`, local courses, professors, rooms, assignments,
  schedule policy snapshot, and constraint policy snapshot.

### 2.1 ManualEditSession

```ts
type ManualEditSession = {
  id: string;
  sourceGenerationId?: string;
  title: string;
  createdAt: string;
  updatedAt: string;

  status: "DRAFT" | "SAVED" | "ARCHIVED";
  version: number;

  baseline: ManualEditSnapshot;
  working: ManualEditSnapshot;

  undoStack: ManualEditSnapshot[];
  redoStack: ManualEditSnapshot[];

  metadata: ManualEditSessionMetadata;
};

type ManualEditSessionMetadata = {
  source: "GENERATED_RESULT" | "SAVED_TIMETABLE_COPY" | "IMPORTED_TIMETABLE";
  createdByUserId?: string;
  lastEditedByUserId?: string;
  notes?: string;
};
```

`baseline` is the snapshot when manual edit started or when the saved version
was last loaded.

`working` is the current edited state.

### 2.2 ManualEditSnapshot

```ts
type ManualEditSnapshot = {
  courses: ManualCourse[];
  professors: ManualProfessor[];
  rooms: ManualRoom[];
  assignments: ManualAssignment[];
  stagedBlocks: StagedBlock[];

  crossGroups: ManualCrossGroup[];
  retakeScenarios: ManualRetakeScenario[];
  schedulePolicySnapshot: SchedulePolicySnapshot;
  constraintPolicySnapshot: ManualConstraintPolicySnapshot;

  displaySettings: ManualDisplaySettings;
};
```

The snapshot contains its own catalog. It must not look up master DB rows at
render or validation time.

### 2.3 Snapshot Catalog Rows

Manual edit catalog rows can start as copies of DB rows, but after session
creation they become independent.

```ts
type ManualCourse = {
  id: string;
  originalCourseId?: string;
  lifecycle: "UNCHANGED" | "ADDED" | "UPDATED" | "DELETED";

  name: string;
  grade: number;
  section: number;
  professorId: string;
  coteachProfs: string[];

  hoursPerWeek: number;
  blockStructure: number[];

  fixedRooms: string[];
  unavailableRooms: string[];

  isFixed: boolean;
  fixedSlots: TimeSlot[];

  isSchoolFixed: boolean;
  schoolFixedTargetGrade?: number | "ALL";
};

type ManualProfessor = {
  id: string;
  originalProfessorId?: string;
  lifecycle: "UNCHANGED" | "ADDED" | "UPDATED" | "DELETED";

  name: string;
  unavailableSlots: TimeSlot[];
  unavailableRooms: string[];
};

type ManualRoom = {
  id: string;
  originalRoomId?: string;
  lifecycle: "UNCHANGED" | "ADDED" | "UPDATED" | "DELETED";

  name: string;
  capacity?: number;
  isLab?: boolean;
};
```

Important:

- `original*Id` is for traceability only.
- manual validation uses the local `id`, not the original DB id.
- deleting a catalog item inside manual edit does not delete it from the DB.

### 2.4 Assignment Rows

Assignments are the persisted truth for timetable placement.

```ts
type ManualAssignment = {
  id: string;                 // stable row id
  occurrenceId: string;       // stable visual block id
  courseId: string;
  day: number;
  period: number;
  roomId: string;
};
```

For a one-room, two-hour block:

```text
2 assignment rows
same occurrenceId
same courseId
same roomId
periods p and p+1
```

For a two-room, two-hour block:

```text
4 assignment rows
same occurrenceId
same courseId
rooms R1 and R2
periods p and p+1
```

The UI renders visual blocks by grouping rows by:

```text
occurrenceId
courseId
day
consecutive periods
room set
```

### 2.5 Staged Blocks

Staged blocks are temporarily removed visual blocks.

```ts
type StagedBlock = {
  id: string;
  occurrenceId: string;
  courseId: string;
  sourceDay: number;
  sourceStartPeriod: number;
  rowSpan: number;
  roomIds: string[];
  assignments: ManualAssignment[];
};
```

Save and export must be blocked while `stagedBlocks.length > 0`.

## 3. Session Lifecycle

### 3.1 Create From Generated Result

When the solver returns a result:

1. Copy generation courses into `ManualCourse[]`.
2. Copy generation professors into `ManualProfessor[]`.
3. Copy generation rooms into `ManualRoom[]`.
4. Copy cross groups and retake scenarios.
5. Copy the schedule policy and constraint settings used by the solver.
6. Convert solver assignments into `ManualAssignment[]`.
7. Set `baseline = working`.

### 3.2 Open Existing Saved Timetable

When opening a saved timetable:

- load the saved manual-edit snapshot.
- do not refresh it from the current DB.
- do not recompute it from current constraints.
- show a small metadata label such as "Snapshot from 2026-07-04 15:20" if the
  product wants context.

### 3.3 Navigate To Constraint Settings

If the user leaves manual edit and opens constraint/data settings:

- settings screen edits master data or future generation inputs.
- returning to the manual edit screen must restore the same manual session.
- changed constraints do not modify `working`.

If the product offers "re-generate with changed constraints", that creates a
new generated result and then a new manual-edit session.

### 3.4 Save

Recommended save behavior:

- `Save Draft`: update the current `ManualEditSession.working`.
- `Save As New Copy`: create a new session/version from `working`.
- `Export`: export `working` without changing master data.

Do not write manual catalog changes into master course/professor/room tables
unless there is a separate explicit command:

```text
Apply manual catalog changes to master data
```

That command is outside the default manual-edit workflow.

## 4. Manual Edit Operations

Manual edit must support the following operation families.

### 4.1 Move Existing Block

Move one visual block to a new day/start period.

Rules:

- move all rows in the occurrence together.
- preserve `occurrenceId`.
- preserve room set unless the user also changes rooms.
- preserve row span.
- validate target period range.
- validate lunch/schedule policy from the session snapshot, not current global
  settings.
- validate conflicts against the local `working` snapshot.

### 4.2 Swap Blocks

Swap two visual blocks.

Rules:

- both blocks must have compatible duration unless product explicitly supports
  reshaping.
- swap all rows in both occurrences.
- preserve each occurrence's room set unless the user also changes rooms.
- validate final timetable after swap.

### 4.3 Stage Block

Remove a visual block from the grid into the staging panel.

Rules:

- remove all assignment rows for the occurrence from `working.assignments`.
- add a `StagedBlock` containing the removed rows.
- save/export is blocked until the block is placed back or deleted through an
  explicit course/assignment deletion flow.
- undo/redo includes staged blocks.

### 4.4 Place Staged Block

Place a staged block into an empty valid slot.

Rules:

- recreate assignment rows from `StagedBlock.assignments`.
- update day and period offsets.
- preserve course id, occurrence id, rooms, and duration.
- validate target conflicts.
- remove the staged block after placement.

### 4.5 Change Rooms For an Existing Block

Change room assignment for one occurrence.

Supported cases:

1. One-room block -> another one-room block.
2. One-room block -> multi-room block.
3. Multi-room block -> one-room block.
4. Multi-room block -> another multi-room set.

Rules:

- update assignment rows for every period in the occurrence.
- if selecting N rooms for a block with duration D, create `N * D` assignment
  rows.
- all rooms in a multi-room occurrence are simultaneous, not alternatives.
- validate room existence in the session local room catalog.
- validate room conflicts.
- if the course has `fixedRooms`, changing occurrence rooms should either:
  - update `course.fixedRooms` for the manual snapshot, or
  - be blocked unless the user edits the course room setting first.

Recommendation: expose this as "Edit course rooms" for fixed-room courses and
"Change this occurrence room" for auto-room/manual override courses.

### 4.6 Edit Existing Course

Manual edit must allow editing a course inside the session snapshot.

Editable fields:

- course name.
- grade.
- section.
- hours per week.
- block structure.
- main professor.
- co-teaching professors.
- fixed rooms.
- unavailable rooms.
- school-fixed fields if product keeps school fixed items editable.

Rules:

- changes apply only to the manual snapshot.
- update all visual labels immediately.
- update validation immediately.
- do not re-run the solver automatically.
- changing `hoursPerWeek` or `blockStructure` may make existing assignments
  invalid. Show validation errors until the user fixes placements.
- changing grade may move the course to a different grade view but must not
  automatically move time slots.
- changing main professor or team-teaching professors changes professor
  conflict validation.
- changing fixed rooms should update assignment rows or require user choice.

Recommended UX for fixed rooms:

When fixed rooms change:

```text
Option A: Apply new rooms to all existing occurrences of this course.
Option B: Keep existing occurrence rooms as manual overrides.
```

For a simpler first version, use Option A only.

### 4.7 Add New Course

Manual edit must allow adding a course that did not exist in the generated
result.

Input fields:

- course id or generated local id.
- course name.
- grade.
- section.
- hours per week.
- block structure.
- main professor.
- co-teaching professors.
- fixed rooms or auto room.
- unavailable rooms.

After adding the course catalog row, the user must place its required blocks.

Recommended flow:

1. User creates course.
2. System creates one or more unplaced `StagedBlock` items based on
   `blockStructure`.
3. User drags or clicks staged blocks into the timetable.
4. Save/export remains blocked until all required blocks are placed.

Generated ids:

```text
manual-course-{sessionId}-{sequence}
manual-professor-{sessionId}-{sequence}
manual-room-{sessionId}-{sequence}
```

### 4.8 Delete Course

Deleting a course inside manual edit is a session-local deletion.

Rules:

- remove or mark the course as `DELETED`.
- remove all assignments for the course.
- remove staged blocks for the course.
- remove the course from cross/retake references or mark those references
  invalid.
- do not delete the course from master DB.

Recommended behavior:

- If the user deletes a course, mark it `DELETED` for audit but hide it from
  normal views.
- On save, keep deleted rows in session history if versioning/audit is needed.

### 4.9 Add or Edit Professor

Manual edit must allow local professor changes because course editing needs
main professor and team-teaching updates.

Operations:

- add manual professor.
- edit professor name.
- edit unavailable slots.
- edit unavailable rooms.
- mark professor deleted if no local course references it.

Rules:

- adding/editing a professor changes only the session catalog.
- professor conflicts use local professor ids.
- team-teaching conflicts include all `coteachProfs`.
- deleting a professor referenced by a course is blocked unless the user first
  changes the course professor/team-teaching fields.

### 4.10 Team-Teaching Changes

A course can have:

```text
main professorId
coteachProfs[]
```

Validation:

- same professor id must not appear twice.
- main professor must not also be in `coteachProfs`.
- every professor id must exist in the session professor catalog.
- professor overlap and unavailable-time validation applies to main and
  co-teaching professors.

Display:

- course card may show main professor first.
- co-teaching professors may follow with comma-separated labels.

### 4.11 Add or Edit Room

Manual edit must allow local room changes.

Operations:

- add room.
- edit room name/capacity/lab metadata.
- delete room if not referenced.
- assign one or more rooms to an occurrence or course fixed-room setting.

Rules:

- local room changes do not modify master DB.
- deleting a room referenced by any assignment, fixed room, or unavailable room
  is blocked unless references are removed first.
- room conflict validation uses session rooms and assignments.

### 4.12 Multi-Room Courses

A multi-room course uses multiple rooms at the same time.

Rules:

- `fixedRooms = ["R1", "R2"]` means R1 and R2 are both occupied for every
  assigned period.
- it does not mean "choose R1 or R2".
- visual cards should show all rooms.
- conflict detection must check each room row separately.
- editing from N rooms to M rooms changes assignment row count.

## 5. Validation Philosophy

Manual edit validation is not the same as generation.

Generation validation asks:

```text
Can the solver create a valid timetable from these constraints?
```

Manual edit validation asks:

```text
Is this edited timetable internally valid as a saved timetable snapshot?
```

Therefore:

- do not automatically enforce newly edited global constraints.
- do not re-solve after every manual edit.
- do validate real conflicts and structural integrity.
- do show warnings for policy preferences if helpful.

### 5.1 How Validation Works Without Live DB Dependency

Manual edit is disconnected from the live master DB, but it is not disconnected
from validation data.

The solution is that `ManualEditSnapshot` must contain every piece of data
needed to validate the edited timetable:

- local courses.
- local professors.
- local rooms.
- local cross groups.
- local retake scenarios.
- generation-time schedule policy.
- generation-time constraint policy.
- current assignment rows.
- staged blocks.

In other words, the manual-edit snapshot becomes the validation database for
that timetable copy.

Manual validation must query:

```text
ManualEditSession.working
```

It must not query:

```text
current master Course table
current master Professor table
current master Room table
current global constraint settings
```

This still allows all important violation checks:

| Violation check | Data source inside snapshot |
| --- | --- |
| Room double-booking | `working.assignments` + `working.rooms` |
| Professor double-booking | `working.assignments` + `working.courses` + `working.professors` |
| Team-teaching conflict | `course.professorId` + `course.coteachProfs` from `working.courses` |
| Professor unavailable time | `working.professors[].unavailableSlots` |
| Professor unavailable room | `working.professors[].unavailableRooms` |
| Course fixed rooms | `working.courses[].fixedRooms` |
| Course unavailable rooms | `working.courses[].unavailableRooms` |
| Multi-room occupancy | assignment rows grouped by `occurrenceId` and room set |
| Same grade overlap | `working.courses[].grade` + `working.assignments` |
| Same base-course section overlap | `working.courses[].id/section` + `working.assignments` |
| Cross/retake rules | `working.crossGroups` + `working.retakeScenarios` |
| Lunch/time-band rules | `working.schedulePolicySnapshot` |
| Required hours/block shape | `working.courses[].hoursPerWeek/blockStructure` + assignments |

If the user edits a course, professor, team-teaching professor list, room, or
multi-room setting inside manual edit, that edit updates the local snapshot
catalog first. Validation then runs against the updated local snapshot.

Example:

```text
1. Generated timetable has Course A taught by Professor P1.
2. User changes Course A to Professor P2 inside manual edit.
3. The master DB is not changed.
4. working.courses[A].professorId becomes P2.
5. Professor conflict validation now uses P2.
```

The only time the master DB may be consulted is through an explicit import
action, such as "copy professor from master data into this timetable". Once
copied, the row becomes local snapshot data and no longer tracks future DB
changes.

Server-side save/export validation must follow the same rule. The save request
must include or load the full manual-edit snapshot, then validate that snapshot
without refreshing labels or constraints from master DB.

## 6. Manual Edit Validation Rules

Validation uses `working` only.

### 6.1 Structural Validation

Check:

- every assignment references an existing non-deleted course.
- every assignment references an existing non-deleted room.
- every course professor id exists.
- every co-teaching professor id exists.
- every visual occurrence has consecutive periods.
- every visual occurrence has consistent room set per period.
- all staged blocks are either placed before save/export or explicitly removed.
- course required hours match assignment hours unless the course is deleted.

Course hours check:

```text
actual assigned time slots for course = distinct(day, period) by course
expected = hoursPerWeek
```

For multi-room courses, do not count room rows as extra hours.

### 6.2 Time Policy Validation

Use the session's `schedulePolicySnapshot`.

Check:

- period exists in snapshot all periods.
- undergraduate course is not in night period.
- graduate course follows the session snapshot's graduate policy.
- lunch policy is respected according to the policy snapshot.
- block does not cross disallowed periods.

Important:

If global settings changed after the session was created, ignore them.

### 6.3 Room Validation

Check:

- same room cannot be used by two different occurrences at same day/period.
- course fixed rooms are respected if fixed-room mode is active.
- course unavailable rooms are not used.
- professor unavailable rooms apply to auto-room/manual rules according to the
  session policy.
- multi-room assignments reserve every listed room.

### 6.4 Professor Validation

Check:

- main professor and co-teaching professors are not double-booked.
- professor unavailable slots are not used.
- deleted professors are not referenced.

### 6.5 Grade and Section Validation

Check:

- same grade overlapping different base courses is an error unless allowed by
  cross/manual cross rules.
- sections of same base course do not overlap.
- changing grade in manual edit revalidates grade conflicts immediately.

### 6.6 Cross and Retake Validation

If the session includes cross groups and retake scenarios:

- keep using the session snapshot definitions.
- if course edit/delete makes a cross group invalid, surface it as a manual edit
  validation error.
- allow user to remove or update the cross group inside manual edit if the web
  product supports that editor.

Recommended first version:

- show cross/retake issues as errors.
- allow removing invalid cross/retake references from the manual session.

### 6.7 Fixed Time and School Fixed Validation

Fixed-time courses:

- must occupy their fixed slots unless the user explicitly turns off fixed
  status in the manual course editor.
- changing fixed slots does not affect master DB.

School fixed rows:

- are part of the session snapshot.
- can remain display-only if product wants them locked.
- if editable, follow the same session-local rules as normal courses.

## 7. Conflict Severity

Use three levels.

```ts
type ManualConflictSeverity = "ERROR" | "WARNING" | "INFO";
```

Recommended mapping:

| Severity | Save allowed? | Examples |
| --- | --- | --- |
| `ERROR` | No | room double-booking, professor double-booking, missing required hours |
| `WARNING` | Yes, with confirmation | soft preference violation, professor day spread |
| `INFO` | Yes | changed from generated snapshot, local-only professor added |

Save/export should be blocked by errors.

## 8. Undo, Redo, Reset

Every successful edit operation must push an undo snapshot.

Undo/redo must include:

- assignments.
- courses.
- professors.
- rooms.
- staged blocks.
- cross/retake local changes.
- display settings that affect edit state if needed.

Reset behavior:

- `Reset to session baseline`: restore `baseline`.
- `Reset current unsaved changes`: restore last saved snapshot.
- Do not reload from DB.

## 9. Save and Export

### 9.1 Save Preconditions

Block save when:

- there are validation errors.
- staged blocks remain.
- assignment rows reference missing/deleted catalog rows.
- required course hours are incomplete.
- multi-room occurrence rows are inconsistent.

Allow save with warnings only after user confirmation.

### 9.2 Save Payload

Persist the whole manual session snapshot.

```ts
type SaveManualEditRequest = {
  sessionId: string;
  expectedVersion: number;
  working: ManualEditSnapshot;
  saveMode: "UPDATE_CURRENT" | "SAVE_AS_COPY";
};
```

Use optimistic concurrency:

- request includes `expectedVersion`.
- if server version changed, reject and ask user to reload or save as copy.

### 9.3 Export

Export uses `working` snapshot.

Export must not look up current DB labels because:

- local course names may differ.
- local professor names may differ.
- local rooms may differ.
- deleted DB rows may still be valid in an old saved timetable snapshot.

## 10. API Boundaries

Recommended API endpoints:

```text
POST /manual-edit/sessions/from-generation/{generationId}
GET  /manual-edit/sessions/{sessionId}
PUT  /manual-edit/sessions/{sessionId}
POST /manual-edit/sessions/{sessionId}/save-as-copy
POST /manual-edit/sessions/{sessionId}/validate
POST /manual-edit/sessions/{sessionId}/export
```

Validation can run client-side for immediate feedback, but server-side
validation is still required before save/export.

## 11. Client State Management

Client should keep:

```ts
type ManualEditClientState = {
  session: ManualEditSession;
  selectedOccurrenceId?: string;
  selectedStagedBlockId?: string;
  activeInspectorTab: "DETAIL" | "COURSE" | "ROOM" | "CONFLICTS";
  dirty: boolean;
  validationResult: ManualValidationResult;
};
```

Do not fetch master DB data while rendering the manual grid.

If the user opens a selector for professor/room/course inside manual edit:

- default list comes from session catalog.
- optional "copy from master data" command may import a DB item into the session
  catalog as a local row.
- imported item becomes independent after import.

## 12. UI Requirements

### 12.1 Page Layout

Recommended areas:

- timetable grid.
- side inspector.
- staged block cabinet.
- conflict panel.
- local catalog editors.

Do not mix global constraint editing controls into this page.

### 12.2 Course Inspector

When a block is selected, show:

- course name.
- grade/section.
- main professor.
- co-teaching professors.
- rooms for selected occurrence.
- duration.
- assignment id/occurrence id for debug mode if needed.

Actions:

- move.
- stage.
- edit course.
- change occurrence room.
- delete occurrence.
- delete course from this timetable.

### 12.3 Local Catalog Editors

Manual edit should include local editors for:

- courses.
- professors.
- rooms.

These editors must be clearly labeled as local to this timetable snapshot.

Example label:

```text
This edit changes only the current timetable copy.
```

### 12.4 Add Course Flow

Recommended UX:

1. Click "Add course".
2. Fill course metadata.
3. Select professor/team-teaching/rooms.
4. Confirm.
5. Required blocks appear in staged block cabinet.
6. User places blocks into the grid.

### 12.5 Delete Course Flow

Recommended UX:

1. Click "Delete from this timetable".
2. Confirm that master data will not be deleted.
3. Remove assignments and staged blocks.
4. Mark course deleted or remove from active catalog.
5. Revalidate.

## 13. Relationship With Constraint Settings

Manual edit keeps a copy of generation-time policies:

```ts
type ManualConstraintPolicySnapshot = {
  hardConstraintModes: unknown;
  softConstraintSettings: unknown;
  createdFromGenerationAt?: string;
};
```

Rules:

- Changes in current constraint settings do not affect the session.
- The manual page may show generation-time policy as read-only context.
- The manual page should not offer "edit constraints" controls directly.
- If user wants new constraints, route to generation flow and create a new
  timetable.

## 14. Relationship With Master DB

### 14.1 Master DB -> Manual Session

Allowed only at session creation or explicit import.

Examples:

- create manual session from generated result.
- import a professor from master DB into local session catalog.
- import a room from master DB into local session catalog.

### 14.2 Manual Session -> Master DB

Default: never.

Optional future command:

```text
Apply selected local catalog changes to master data
```

If implemented, it must be explicit, reviewed, and audited.

## 15. Important Edge Cases

### 15.1 Course Hours Changed

If `hoursPerWeek` changes:

- existing assignments may be too many or too few.
- validation should report the difference.
- system may offer to create/delete staged blocks, but should not silently
  change the timetable.

### 15.2 Block Structure Changed

If `blockStructure` changes:

- existing occurrences may not match required block lengths.
- validation should report invalid block shape.
- user can stage/delete/recreate blocks.

### 15.3 Professor Changed

If main professor or co-teaching professors change:

- update displayed professor names.
- revalidate professor conflicts and unavailable slots.
- do not move assignments automatically.

### 15.4 Room Set Changed

If course fixed rooms change:

- update all occurrences or ask user whether to apply globally.
- revalidate room conflicts.
- for multi-room course, create/remove assignment rows per room.

### 15.5 Deleted Professor or Room

If a local professor/room is referenced:

- block deletion.
- offer "replace references first".

### 15.6 Deleted Course With Cross Group

If deleting a course breaks a cross group:

- remove the cross reference, or
- mark the cross group invalid and block save until resolved.

Recommendation: prompt user to remove broken cross/retake references when
deleting a course.

### 15.7 Saved Timetable Opened After Master Data Changed

Expected behavior:

- show exactly the saved snapshot.
- do not update names/rooms/professors from DB.
- optional info message: "This timetable uses its saved snapshot."

## 16. Required Tests

### Snapshot Independence

- Create session from generated result, then modify master course name.
- Manual session still displays original course name.
- Modify master constraint lunch policy.
- Manual session still validates using original policy snapshot.

### Screen Separation

- Navigate manual edit -> constraint settings -> manual edit.
- Manual edit state is unchanged.
- Generating again creates a new session, not mutation of the existing one.

### Course CRUD

- Add course creates local course and staged required blocks.
- Edit course professor updates labels and conflict detection.
- Edit team-teaching adds professor conflict detection.
- Delete course removes assignments and staged blocks.
- Master DB course list is unchanged.

### Room and Multi-Room

- Change one-room occurrence to two rooms creates correct assignment rows.
- Change two-room occurrence to one room removes extra rows.
- Room conflict is detected for each room in a multi-room occurrence.
- Deleting referenced room is blocked.

### Professor

- Add local professor and assign to course.
- Professor conflict detection uses local professor.
- Deleting referenced professor is blocked.

### Save and Export

- Save blocked with staged blocks.
- Save blocked with missing course hours.
- Save payload includes courses, professors, rooms, assignments, policy
  snapshot.
- Export uses local snapshot labels, not current DB labels.

### Undo and Redo

- Undo course edit restores course metadata and validation.
- Undo room change restores assignment row count.
- Undo course deletion restores assignments.
- Undo staging restores grid rows.
